# AI 에이전트 개념 가볍게 보기

AI 에이전트는 단순히 “질문에 답하는 AI”보다 조금 더 넓은 개념입니다.

사용자의 요청을 받고, 필요한 도구를 사용하고, 중간 결과를 바탕으로 다음 일을 이어서 처리하는 흐름을 가질 수 있습니다.

## 이번 실습의 에이전트 구성

| 에이전트 개념 | 이번 실습에서의 모습 |
| --- | --- |
| Trigger | 프론트에서 `분석하기` 버튼을 누르면 시작됩니다. |
| Tool use | n8n이 Upstage OCR과 Chat API를 호출합니다. |
| Context | 사용자의 노트와 참고자료가 AI에게 전달됩니다. |
| RAG-lite | 참고자료를 프롬프트에 넣어 AI가 근거로 삼게 합니다. |
| Structured output | 피드백은 XML, 퀴즈는 JSON으로 받습니다. |
| Memory | 이번 실습에서는 결과를 저장하지 않습니다. |

## Trigger

Trigger는 “언제 시작할까?”에 대한 답입니다.

이번 실습에서는 사용자가 웹 화면에서 `분석하기` 버튼을 누르면 프론트가 n8n Webhook을 호출합니다. 이 호출이 워크플로우의 시작점입니다.

## Tool use

Tool use는 AI 시스템이 외부 도구를 사용하는 것입니다.

이번 실습에서는 n8n이 다음 도구를 사용합니다.

- Upstage OCR API
- Upstage Chat API

## Context

Context는 AI가 답변할 때 참고하는 정보입니다.

이번 실습에서는 다음이 Context가 됩니다.

- 사용자가 입력한 노트
- 사용자가 붙여넣은 참고자료
- OCR로 읽은 텍스트

## Structured output

AI가 자유롭게 문장을 쓰면 화면에서 처리하기 어렵습니다.

그래서 피드백은 XML, 퀴즈는 JSON이라는 정해진 형식으로 달라고 요청합니다.

```json
[
  {
    "question": "광합성은 빛 에너지를 이용한다.",
    "answer": "O",
    "explanation": "광합성은 빛 에너지를 화학 에너지로 전환한다."
  }
]
```

## Memory

Memory는 이전 대화나 결과를 저장해 두고 나중에 다시 사용하는 기능입니다.

이번 실습에서는 결과를 저장하지 않습니다. 학습 흐름을 단순하게 유지하기 위해서입니다.
