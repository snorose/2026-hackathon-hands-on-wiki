# 프롬프트 만들기

프롬프트는 AI에게 보내는 요청문입니다.

“이 노트를 보고 피드백해 줘”라고만 쓰면 답변 형식이 매번 달라질 수 있습니다.

그래서 n8n은 AI에게 더 구체적으로 요청합니다.

## LLM 노드 이해하기

LLM 노드는 회사에 AI 언어 전문가를 초대하는 것과 비슷합니다. AI는 문장을 이해하고, 요약하고, 분류하고, 답변을 만들 수 있지만, 좋은 결과를 얻으려면 업무 지시가 분명해야 합니다.

대표적으로 OpenAI, Anthropic Claude, Google Gemini, DeepSeek, Qwen, Upstage Solar 같은 여러 LLM 계열을 선택할 수 있습니다. 모델별 성능, 가격, 지원 기능은 자주 바뀌므로 문서에서는 특정 모델이 항상 최고라고 외우기보다 아래 기준으로 고르는 것이 좋습니다.

| 선택 기준 | 확인할 것 |
| --- | --- |
| 언어와 도메인 | 한국어, 교육 자료, 문서 분석에 잘 맞는가 |
| 입력 길이 | 긴 참고자료나 노트를 함께 넣을 수 있는가 |
| 출력 안정성 | JSON, XML 같은 정해진 형식을 잘 지키는가 |
| 비용 | 반복 실습과 운영 비용을 감당할 수 있는가 |
| 연동 방식 | n8n에서 API Key와 HTTP Request로 쉽게 호출할 수 있는가 |

이번 실습에서는 `.env`에 넣은 Upstage API Key를 사용해 Upstage Chat API를 호출합니다. 중요한 것은 “어떤 모델인가”보다 **무엇을 어떻게 지시하는가**입니다.

## AI에게 업무 지시하는 3가지 방법

AI에게는 신입사원에게 업무 매뉴얼을 주듯이 기본 역할, 실제 요청, 세부 설정을 함께 알려 주면 좋습니다.

| 방법 | 의미 | 예시 |
| --- | --- | --- |
| 시스템 프롬프트 | AI의 역할과 업무 스타일을 정합니다. | “당신은 친절한 학습 피드백 선생님입니다.” |
| 유저 프롬프트 | 이번에 처리할 구체적인 입력과 요청입니다. | “다음 노트를 보고 잘못된 개념을 찾아 주세요.” |
| LLM 설정값 | 답변의 길이와 성격을 조정합니다. | temperature, max tokens, top p |

설정값은 AI의 성격을 조절하는 손잡이입니다.

| 설정 | 쉬운 의미 | 사용 팁 |
| --- | --- | --- |
| Temperature | 답변의 다양성 | 형식이 중요한 작업은 낮게, 아이디어 생성은 높게 |
| Max Tokens | 답변 최대 길이 | 너무 짧으면 잘리고, 너무 길면 비용이 커짐 |
| Top P | 선택할 단어 후보의 폭 | 보통 기본값으로 시작 |

이번처럼 XML이나 JSON 형식이 중요한 작업에서는 temperature를 낮게 두는 편이 안정적입니다.

## 좋은 업무 매뉴얼의 4단계

좋은 프롬프트는 보통 아래 네 가지를 포함합니다.

1. 역할 정의: AI가 누구처럼 일해야 하는지 정합니다.
2. 행동 지침: 어떤 기준으로 판단해야 하는지 알려 줍니다.
3. 출력 형식: 어떤 구조로 답해야 하는지 지정합니다.
4. 주의사항: 하지 말아야 할 일을 분명히 적습니다.

예를 들어 고객 상담 자동화라면 이렇게 쓸 수 있습니다.

```text
당신은 우리 회사의 전문 고객 상담원입니다.

행동 지침:
- 항상 친절하고 정중하게 답변합니다.
- 고객의 감정을 먼저 이해하고 공감합니다.
- 확실하지 않은 내용은 추측하지 말고 확인이 필요하다고 말합니다.

답변 형식:
1. 문의 내용 요약
2. 고객 감정 파악
3. 해결 방법 제시
4. 추가 도움 확인

주의사항:
- 개인정보를 요구하지 않습니다.
- 답변은 500자 이내로 작성합니다.
```

## 피드백 프롬프트의 구성

피드백 프롬프트에는 다음 정보가 들어갑니다.

- AI의 역할: 선생님
- 판단 기준: 참고자료가 있으면 우선 참고
- 출력 형식: XML
- 참고자료: `<정보>` 영역
- 사용자의 노트: `<입력>` 영역

## 참고자료 영역

```xml
<정보>
참고자료 내용
</정보>
```

이 영역은 AI가 피드백할 때 기준으로 삼는 자료입니다.

## 입력 영역

```xml
<입력>
사용자가 작성한 노트
</입력>
```

이 영역은 실제로 검사할 노트입니다.

## 왜 형식을 정하나요?

프론트 화면은 AI 답변을 읽어서 예쁘게 보여줘야 합니다.

답변 형식이 매번 다르면 화면에서 처리하기 어렵습니다. 그래서 XML, JSON처럼 정해진 형식을 요청합니다.

## 고급 프롬프트 기법

실무에서는 아래 기법을 함께 사용하면 결과가 더 안정적입니다.

| 기법 | 의미 | 언제 유용한가요? |
| --- | --- | --- |
| Format Specification | JSON, XML 같은 출력 형식을 정확히 지정 | 프론트 화면이나 다음 노드가 결과를 파싱해야 할 때 |
| Teaching by Examples | 입력과 출력 예시를 함께 제공 | 원하는 답변 스타일을 AI에게 보여 주고 싶을 때 |
| Spec Files | 프로젝트 목표, 브랜드 톤, 제약사항을 문서로 제공 | 여러 워크플로우가 같은 기준을 공유해야 할 때 |
| Step-by-step Checklist | 판단 순서를 체크리스트로 제공 | 분류, 우선순위 판단, 복잡한 분석이 필요할 때 |

### Format Specification 예시

고객 피드백을 분석한다면 출력 형식을 이렇게 정할 수 있습니다.

```json
{
  "sentiment": "긍정/중립/부정",
  "category": "제품/서비스/배송/기타",
  "priority": "높음/중간/낮음",
  "key_issues": ["이슈1", "이슈2"],
  "suggested_actions": ["액션1", "액션2"],
  "follow_up_required": true
}
```

프롬프트에는 “위 JSON 형식으로만 답변해 주세요”처럼 명확히 적습니다.

### Teaching by Examples 예시

```text
예시 입력:
배송이 너무 늦어서 실망했습니다.

예시 출력:
{
  "sentiment": "부정",
  "category": "배송",
  "priority": "중간",
  "key_issues": ["배송 지연"],
  "suggested_actions": ["배송 상태 확인", "사과 메시지 발송"],
  "follow_up_required": true
}

이제 다음 피드백을 같은 기준으로 분석하세요:
{{ $json.feedback }}
```

예시는 AI에게 “이런 식으로 일하면 된다”는 모범 답안을 보여 주는 역할을 합니다.

### Spec Files 예시

프로젝트 기준이 길다면 별도 문서처럼 정리해 프롬프트에 넣을 수 있습니다.

```md
# 프로젝트 명세서

## 목표
- 고객 만족도 향상
- 응답 시간 단축

## 브랜드 가이드라인
- 톤: 친근하고 전문적
- 스타일: 간결하고 명확

## 제약사항
- 개인정보 수집 금지
- 500자 이내 답변
```

이런 기준은 여러 노드에서 반복해서 사용할 수 있어 응답 품질을 일정하게 유지하는 데 도움이 됩니다.

### Step-by-step Checklist 예시

AI에게 복잡한 판단을 맡길 때는 내부적으로 확인할 순서를 적어 주면 좋습니다.

```text
고객 문의를 처리할 때 아래 순서로 판단하세요.

1. 문의 내용을 한 문장으로 요약합니다.
2. 문의 유형을 분류합니다.
3. 고객 감정 상태를 파악합니다.
4. 우선순위를 정합니다.
5. 해결 방법과 후속 조치를 제안합니다.

최종 답변에는 요약, 분류, 우선순위, 응답문만 포함하세요.
```

:::tip
단계별 판단 기준을 주는 것은 좋지만, 최종 사용자에게 불필요하게 긴 추론 과정을 그대로 보여 줄 필요는 없습니다. 최종 출력에는 필요한 결론과 근거만 남기는 편이 안전합니다.
:::

## 실전 통합 예시

고객이 “주문한 제품이 불량품으로 도착했고 즉시 환불을 원한다”고 문의했다고 가정해 보겠습니다.

이때 좋은 프롬프트는 여러 기법을 함께 사용합니다.

| 기법 | 이 사례에서 하는 일 |
| --- | --- |
| Spec Files | 진심 어린 사과, 개인정보 미요구, 500자 이내 같은 기준을 제공합니다. |
| Step-by-step Checklist | 불만 유형, 긴급도, 감정 상태, 해결책을 순서대로 판단하게 합니다. |
| Teaching by Examples | 비슷한 불만 처리 성공 사례를 보여 줍니다. |
| Format Specification | 분석 결과와 후속 조치를 JSON으로 정리하게 합니다. |

예상 출력은 다음처럼 구조화할 수 있습니다.

```json
{
  "analysis": {
    "category": "제품 불량",
    "emotion": "분노",
    "priority": "최우선",
    "solution": "즉시 환불 및 보상 안내"
  },
  "response": "불량품으로 불편을 드려 진심으로 죄송합니다. 즉시 환불 절차를 안내드리고, 재발 방지를 위해 품질 확인을 진행하겠습니다.",
  "next_actions": ["환불 절차 안내", "수거 방법 안내", "품질관리팀 공유"]
}
```

이번 Solar Teacher 실습의 피드백과 퀴즈 생성도 같은 원리입니다. 역할을 정하고, 참고자료와 입력을 구분하고, 출력 형식을 지정해야 프론트 화면이 결과를 안정적으로 보여 줄 수 있습니다.
